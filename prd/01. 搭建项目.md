# 01. 搭建项目 PRD

## 1. 背景

企业级协同知识库用于帮助团队沉淀、编辑、协作和检索组织内部知识。项目核心能力围绕文档编辑展开，并逐步扩展到协同编辑、RAG 知识库查询和文档内 AI 助手。

完整企业级知识库系统通常包含登录注册、权限系统、用户管理、文档管理、知识库管理等大量基础业务能力。当前版本优先建设核心技术链路和工程底座，基础业务能力暂不纳入本阶段范围。

项目核心技术链路包括：

1. 富文本编辑
2. 多用户协同编辑
3. RAG 知识库查询
4. 文档内 AI 助手

因此，“搭建项目”阶段的目标不是完成完整业务闭环，而是先搭建一个能够承载后续核心模块的工程架构。

## 2. 目标

### 2.1 业务目标

搭建企业级协同知识库项目的基础工程，使富文本编辑、协同编辑、RAG 知识库查询和文档内 AI 助手能力可以在同一套工程体系内持续演进。

### 2.2 技术目标

1. 使用 monorepo 管理多应用和多领域包。
2. 使用领域驱动的目录组织方式拆分核心模块。
3. 搭建 Next.js 主应用，用于承载知识库前端页面和编辑器体验。
4. 预留协同服务应用，用于后续接入 Yjs / Hocuspocus。
5. 预留编辑器、协同、RAG、AI、数据库、共享能力等领域包。
6. 建立统一的 TypeScript、代码规范和包管理基础配置。

## 3. 范围

### 3.1 本阶段包含

1. 初始化项目根目录。
2. 配置 pnpm workspace。
3. 创建 monorepo 基础目录结构。
4. 创建 `apps/web` Next.js 主应用。
5. 创建 `apps/collab` 协同服务应用占位。
6. 创建领域包目录：
   - `packages/editor`
   - `packages/ai-workspace`
   - `packages/rag`
   - `packages/collaboration`
   - `packages/auth`
   - `packages/database`
   - `packages/shared`
7. 创建文档、基础设施、脚本目录。
8. 配置根级 `package.json`、`pnpm-workspace.yaml`、`tsconfig.base.json`、`.gitignore`。
9. 确保项目可以安装依赖、启动主应用、通过基础类型检查。

### 3.2 本阶段不包含

1. 不实现登录注册。
2. 不实现权限系统。
3. 不实现用户管理。
4. 不实现文档管理。
5. 不实现知识库管理。
6. 不实现完整富文本编辑器。
7. 不实现多人协同编辑。
8. 不实现 RAG 检索和问答。
9. 不实现 AI 助手交互。
10. 不接入真实数据库业务模型。

以上能力会在后续阶段逐步实现。本阶段只负责搭建可扩展的工程底座。

## 4. 用户与使用场景

### 4.1 目标用户

本阶段面向项目开发团队和后续模块维护者。

开发者需要通过该阶段明确：

1. 项目采用 monorepo 的工程组织方式。
2. 项目核心模块按照业务领域拆分，而不是只按技术类型拆分。
3. 各应用和领域包的职责边界。
4. 后续富文本、协同、RAG、AI 能力分别应该落在哪些工程边界内。

### 4.2 核心场景

开发者拉取项目后，可以执行依赖安装命令，并启动主应用，看到一个基础页面。开发者也可以从目录结构上清楚判断：

1. 前端页面在哪里开发。
2. 协同服务在哪里开发。
3. 编辑器能力放在哪个包。
4. AI 助手能力放在哪个包。
5. RAG 能力放在哪个包。
6. 通用类型和工具方法放在哪个包。

## 5. 产品需求

### 5.1 项目初始化

项目名称建议为 `ai-doc-platform`。

项目根目录需要包含 monorepo 的基础配置文件，并通过 pnpm workspace 管理 `apps/*` 和 `packages/*`。

验收标准：

1. 根目录存在 `package.json`。
2. 根目录存在 `pnpm-workspace.yaml`。
3. 根目录存在 `tsconfig.base.json`。
4. 根目录存在 `.gitignore`。
5. 执行 `pnpm install` 可以正常安装依赖。

### 5.2 应用目录

项目需要包含两个应用：

```text
apps/
├── web/      # Next.js 主应用
└── collab/   # Yjs / Hocuspocus 协同服务
```

`apps/web` 是用户主要访问的前端应用，后续富文本编辑器、AI 助手入口、RAG 查询入口都会先在这里集成。

`apps/collab` 是协同服务应用，后续用于启动 WebSocket 服务，并接入 Yjs / Hocuspocus。

验收标准：

1. `apps/web` 可以通过命令启动。
2. `apps/web` 至少包含一个首页页面。
3. `apps/collab` 目录存在，并包含独立的包配置或服务入口占位。
4. 两个应用的职责在目录和命名上清晰可识别。

### 5.3 领域包目录

项目需要按照领域驱动方式组织核心能力，而不是只按照 `components`、`hooks`、`utils`、`services` 这类技术类型拆分。

需要创建以下包：

```text
packages/
├── editor/          # BlockNote 编辑器领域
├── ai-workspace/    # AI 助手和文档内 AI 交互领域
├── rag/             # RAG 领域，后续基于 LlamaIndex
├── collaboration/   # 协同编辑领域
├── auth/            # 权限领域占位
├── database/        # Prisma / Postgres / pgvector 数据库能力
└── shared/          # 通用类型、工具、配置
```

各包职责如下：

| 包名 | 职责 |
| :--- | :--- |
| `editor` | 封装富文本编辑器能力，后续接入 BlockNote |
| `ai-workspace` | 封装 AI 润色、改写、续写等文档内 AI 能力 |
| `rag` | 封装文档索引、检索、问答等 RAG 能力 |
| `collaboration` | 封装协同编辑相关类型、协议和客户端适配 |
| `auth` | 预留 RBAC / ACL 等权限能力 |
| `database` | 统一管理数据库访问、ORM、向量存储相关能力 |
| `shared` | 存放跨领域复用的类型、工具函数和基础配置 |

验收标准：

1. 每个领域包都有独立目录。
2. 每个领域包至少包含基础入口文件。
3. 每个领域包的命名能反映业务领域。
4. `shared` 只承载通用能力，不承载具体业务流程。

### 5.4 文档目录

项目需要预留工程文档目录，用于沉淀架构、编辑器、协同、RAG、AI、权限等主题文档。

目录结构如下：

```text
docs/
├── architecture/
├── editor/
├── rag/
├── ai/
├── collaboration/
└── auth/
```

验收标准：

1. `docs` 目录存在。
2. 各主题目录存在。
3. 后续模块文档有明确归属位置。

### 5.5 基础设施目录

项目需要预留基础设施目录，用于后续放置 Docker、Nginx、监控、Kubernetes 等部署相关配置。

目录结构如下：

```text
infrastructure/
├── docker/
├── nginx/
├── monitoring/
└── kubernetes/
```

验收标准：

1. `infrastructure` 目录存在。
2. 各基础设施子目录存在。
3. 本阶段不要求提供完整部署配置。

### 5.6 脚本目录

项目需要创建 `scripts` 目录，用于后续放置项目初始化、数据处理、构建辅助、索引任务等脚本。

验收标准：

1. `scripts` 目录存在。
2. 本阶段不要求实现具体脚本。

## 6. 工程需求

### 6.1 包管理

使用 pnpm 作为包管理工具。

根目录 `pnpm-workspace.yaml` 需要包含：

```yaml
packages:
  - "apps/*"
  - "packages/*"
```

根目录 `package.json` 需要提供基础脚本：

```json
{
  "scripts": {
    "dev": "pnpm --filter web dev",
    "build": "pnpm -r build",
    "typecheck": "pnpm -r typecheck",
    "lint": "pnpm -r lint"
  }
}
```

具体脚本可根据实际项目依赖调整，但需要保证根目录可以统一调度子应用和子包。

### 6.2 TypeScript 配置

项目需要提供根级 `tsconfig.base.json`，用于统一 TypeScript 基础配置。

推荐要求：

1. 开启严格类型检查。
2. 统一模块解析策略。
3. 统一路径别名基础规则。
4. 子应用和子包继承根配置。

### 6.3 代码规范

本阶段至少需要预留 lint 能力。

推荐要求：

1. `apps/web` 可使用 Next.js 默认 ESLint 配置。
2. 后续根据项目需要再统一扩展 monorepo 级 lint 配置。

### 6.4 环境变量

根目录可以创建 `.env.example`，用于描述后续需要的环境变量。

本阶段可以预留：

```text
DATABASE_URL=
OPENAI_API_KEY=
COLLAB_SERVER_URL=
```

验收标准：

1. 不提交真实密钥。
2. `.gitignore` 忽略 `.env`。
3. `.env.example` 可以说明后续配置项。

## 8. 页面需求

`apps/web` 首页只需要作为项目启动验证页面，不要求实现完整业务。

首页建议展示以下信息：

1. 项目名称：企业级协同知识库。
2. 当前阶段：项目搭建。
3. 后续核心模块：
   - 富文本编辑
   - 多人协同编辑
   - RAG 知识库查询
   - 文档内 AI 助手

验收标准：

1. 启动 `apps/web` 后可以访问首页。
2. 页面能证明 Next.js 应用已正常运行。
3. 页面不需要设计复杂交互。

## 9. 非功能需求

### 9.1 可维护性

项目结构需要让领域边界清晰。后续新增功能时，应优先判断该功能属于哪个领域包，而不是默认放入主应用。

### 9.2 可扩展性

项目需要支持后续接入：

1. BlockNote 编辑器。
2. Yjs 协同数据结构。
3. Hocuspocus WebSocket 服务。
4. LlamaIndex RAG 能力。
5. Prisma / Postgres / pgvector 数据库能力。
6. AI 模型调用能力。

### 9.3 可运行性

本阶段完成后，项目至少需要满足：

1. 可以安装依赖。
2. 可以启动前端应用。
3. 可以执行基础类型检查。
4. 目录结构可以承载后续模块实现。

## 11. 任务拆分

### 11.1 初始化根工程

1. 创建项目目录。
2. 初始化根 `package.json`。
3. 配置 `pnpm-workspace.yaml`。
4. 配置 `tsconfig.base.json`。
5. 配置 `.gitignore`。
6. 创建 `.env.example`。

### 11.2 创建应用

1. 创建 `apps/web`。
2. 初始化 Next.js 应用。
3. 创建首页启动验证页面。
4. 创建 `apps/collab`。
5. 添加协同服务入口占位。

### 11.3 创建领域包

1. 创建 `packages/editor`。
2. 创建 `packages/ai-workspace`。
3. 创建 `packages/rag`。
4. 创建 `packages/collaboration`。
5. 创建 `packages/auth`。
6. 创建 `packages/database`。
7. 创建 `packages/shared`。
8. 为各包添加基础入口文件和包配置。

### 11.4 创建支撑目录

1. 创建 `docs` 及主题子目录。
2. 创建 `infrastructure` 及子目录。
3. 创建 `scripts` 目录。

### 11.5 验证

1. 执行依赖安装。
2. 启动前端应用。
3. 执行类型检查。
4. 检查目录结构是否符合本 PRD。

## 12. 后续衔接

本阶段完成后，项目具备后续模块开发的工程基础。

后续阶段建议按以下顺序推进：

1. 富文本基础编辑：在 `packages/editor` 中封装编辑器能力，并在 `apps/web` 中集成。
2. 多用户协同编辑：在 `packages/collaboration` 和 `apps/collab` 中实现协同链路。
3. RAG 知识库查询：在 `packages/rag` 和 `packages/database` 中实现索引、检索和问答能力。
4. AI 助手：在 `packages/ai-workspace` 中实现润色、改写、续写，并和编辑器选区能力打通。

# 03. editor 需求说明书

## 1. 背景

在 `01. 搭建项目` 阶段中，项目已经创建了 `packages/editor` 领域包，并在 `apps/web` 中预留了文档编辑页面。

在 `02. 书写静态页面` 阶段中，项目已经完成工作台静态页面，包括文档编辑页、协同状态条、AI Assistant 面板和 RAG 检索工作台。当前 editor 阶段中，文档编辑页以编辑器和文档状态面板为核心，不再要求在文档页右侧保留 AI Assistant 面板。当前文档编辑页仍然以 mock 数据为基础，占位组件位于 `apps/web/src/components/editor/editor-shell.tsx`，文档数据来自 `apps/web/src/mock/workspace.ts` 和本地 mock 文档状态。

本需求针对 `packages/editor` 从占位包演进为真实 Editor Adapter 的阶段，目标是让它承接 BlockNote 富文本编辑能力，并通过稳定接口提供给 `apps/web` 使用。

经过技术调研，BlockNote 已经内置较完整的富文本编辑能力，并提供 AI 编辑扩展和 Yjs 协同接入能力。因此，本阶段不应在项目中自研富文本编辑器内核，也不应新增一个泛化的 `adapter` 包来承载第三方依赖。

本阶段应将 `packages/editor` 明确定位为项目级 Editor Adapter：以 BlockNote 作为底层编辑器能力，封装为 `ai-doc-platform` 自己的文档编辑器接口，供 `apps/web` 使用。

## 2. 目标

### 2.1 业务目标

将当前静态文档编辑页升级为可编辑的富文本文档区域，使用户能够在工作台内完成基础文档编辑操作。

用户进入 `/docs/[id]` 页面后，应能看到：

1. 文档标题、作者、状态等现有页面信息仍然保留。
2. 中间主区域从静态内容替换为真实 BlockNote 编辑器。
3. 用户可以输入文字、创建标题、列表、引用、代码块等基础内容。
4. 用户可以通过 slash menu 使用 BlockNote 基础块插入能力。
5. 用户可以选中文本并看到基础格式化工具栏。
6. 编辑器区域仍然与顶部 PresenceBar、文档状态面板保持现有布局关系。

### 2.2 技术目标

1. 将 `packages/editor` 从占位包改造为 BlockNote Adapter 包。
2. 在 `packages/editor` 中集中管理 `@blocknote/*` 相关依赖和初始化逻辑。
3. 避免在 `apps/web` 页面和业务组件中散落 BlockNote 细节。
4. 在 `packages/editor` 中导出项目自己的 `DocumentEditor` 组件和编辑器类型。
5. 在 `packages/editor` 中定义项目级文档内容类型，降低业务层对 BlockNote 类型的直接依赖。
6. 在 `apps/web` 文档页中替换静态编辑器内容，但保留当前工作台布局、视觉层级和 mock 数据结构。
7. 为后续接入 `packages/ai-workspace`、`packages/collaboration` 和真实文档保存能力预留清晰接口。

## 3. 范围

### 3.1 本阶段包含

1. 在 `packages/editor` 中接入 BlockNote 基础依赖。
2. 在 `packages/editor` 中创建 BlockNote Adapter 目录结构。
3. 在 `packages/editor` 中实现 `DocumentEditor` 组件。
4. 在 `packages/editor` 中封装 BlockNote 初始化参数。
5. 在 `packages/editor` 中提供初始文档内容配置。
6. 在 `packages/editor` 中定义项目级编辑器内容类型。
7. 在 `packages/editor` 中提供基础内容转换函数。
8. 在 `apps/web` 中把 `EditorShell` 的静态正文区域替换为真实编辑器挂载区。
9. 保持文档页现有布局：PresenceBar 在上方，编辑器为主区域，文档状态面板用于展示保存状态和文档操作。
10. 保持 mock 数据驱动，不接入真实数据库。
11. 保证编辑器在 Next.js App Router 环境下可运行，处理好客户端组件和 SSR 边界。
12. 保证项目可以通过基础类型检查。

### 3.2 本阶段不包含

1. 不自研富文本编辑器内核。
2. 不新增通用 `packages/adapter` 包。
3. 不实现真实文档 CRUD。
4. 不接入数据库保存。
5. 不实现自动保存。
6. 不实现版本历史。
7. 不实现评论、批注和审阅流。
8. 不实现真实多人协同编辑。
9. 不接入 Yjs provider。
10. 不接入 Hocuspocus 服务。
11. 不实现真实 AI 模型调用。
12. 不实现 BlockNote AI Extension 的完整服务端链路。
13. 不实现 RAG 检索结果插入文档。
14. 不实现复杂自定义 block schema。
15. 不实现移动端完整编辑体验优化。

本阶段只负责把静态编辑区升级为真实 BlockNote 编辑器，并建立 `packages/editor` 的长期封装边界。

## 4. 用户与使用场景

### 4.1 目标用户

本阶段主要面向项目开发者、课程学习者和后续模块维护者。

开发者需要通过本阶段明确：

1. 项目不会自研编辑器底层能力。
2. BlockNote 是当前项目的富文本编辑器底座。
3. `packages/editor` 是 BlockNote 的项目级适配层。
4. `apps/web` 不直接承担编辑器内核集成细节。
5. AI、协同、RAG 与编辑器的关系通过领域包边界逐步接入。

### 4.2 核心场景

用户访问文档编辑页后，可以完成以下操作：

1. 查看当前文档的标题、作者、状态和协同状态占位。
2. 在编辑器中输入和编辑正文内容。
3. 使用 `/` 打开 BlockNote slash menu。
4. 插入基础块，如标题、列表、引用、代码块等。
5. 选中文本后使用格式化工具栏。
6. 在页面中继续看到文档状态面板，包括保存状态、草稿状态和保存操作。

## 5. 架构定位

### 5.1 包职责

`packages/editor` 的职责是封装文档编辑器领域能力，而不是提供泛化第三方 SDK 适配能力。

推荐边界如下：

```text
apps/web
  负责页面组合、路由、布局和业务数据传入

packages/editor
  负责 BlockNote Adapter、编辑器组件、内容类型、转换函数和编辑器命令接口

packages/ai-workspace
  后续负责 AI 动作、提示词、RAG 上下文和文档内 AI 工作流

packages/collaboration
  后续负责 Yjs provider、协同房间、用户 presence 和同步状态
```

### 5.2 不新增 adapter 包的原因

本阶段不新增 `packages/adapter` 或 `packages/blocknote-adapter`。

原因：

1. `adapter` 命名过于泛化，容易变成第三方依赖杂物包。
2. BlockNote 不是横向基础设施，而是编辑器领域的底层实现。
3. 当前项目规模下，第三方库适配逻辑放在对应领域包内更清晰。
4. `packages/editor` 已经存在，并且正是承载编辑器适配逻辑的合适位置。

## 6. 推荐目录结构

`packages/editor` 推荐调整为：

```text
packages/editor
├── package.json
├── tsconfig.json
└── src
    ├── blocknote
    │   ├── document-editor.tsx
    │   ├── create-editor-options.ts
    │   ├── initial-content.ts
    │   ├── serialization.ts
    │   └── styles.css
    ├── commands.ts
    ├── types.ts
    └── index.ts
```

说明：

1. `blocknote/document-editor.tsx` 封装真实 BlockNote 组件。
2. `blocknote/create-editor-options.ts` 管理 BlockNote 初始化参数。
3. `blocknote/initial-content.ts` 管理本阶段默认文档内容。
4. `blocknote/serialization.ts` 管理项目内容类型与 BlockNote blocks 的转换。
5. `commands.ts` 定义未来可暴露给外部的编辑器命令接口。
6. `types.ts` 定义项目级编辑器内容类型。
7. `index.ts` 作为对外唯一出口。

`apps/web` 推荐继续保留当前页面结构：

```text
apps/web/src/app/(workspace)/docs/[id]/page.tsx
apps/web/src/components/editor/editor-shell.tsx
```

其中 `editor-shell.tsx` 负责页面容器、文档元信息和视觉布局，真实编辑器组件由 `packages/editor` 提供。

## 7. 产品需求

### 7.1 文档编辑器组件

`packages/editor` 需要提供 `DocumentEditor` 组件。

推荐使用方式：

```tsx
<DocumentEditor
  documentId={document.id}
  initialContent={content}
  editable={true}
  onChange={handleChange}
/>
```

组件能力：

1. 支持传入文档 ID。
2. 支持传入初始内容。
3. 支持控制是否可编辑。
4. 支持内容变化回调。
5. 支持基础样式适配。
6. 支持在 Next.js 客户端组件中使用。

验收标准：

1. `packages/editor` 对外导出 `DocumentEditor`。
2. `apps/web` 可以使用 `DocumentEditor` 替换静态正文区域。
3. 编辑器可以正常输入文本。
4. 编辑器可以正常展示初始内容。
5. 页面刷新后不要求保留用户修改内容。

### 7.2 基础富文本编辑

编辑器需要支持 BlockNote 默认富文本能力。

本阶段至少需要覆盖：

1. 段落。
2. 一级、二级、三级标题。
3. 有序列表。
4. 无序列表。
5. 引用块。
6. 代码块。
7. 文本加粗。
8. 文本斜体。
9. 文本删除线。
10. 链接。

验收标准：

1. 用户可以直接输入正文。
2. 用户可以通过 slash menu 插入常见块。
3. 用户可以选中文本并使用基础格式化工具栏。
4. 编辑器不应破坏当前文档页布局。

### 7.3 初始内容

当前项目还没有真实文档内容字段，因此本阶段可以继续使用 mock 初始内容。

推荐策略：

1. 在 `packages/editor` 中提供默认 `initialContent`。
2. `apps/web` 可以先不扩展 mock 文档正文结构。
3. 如需表达不同文档内容，可以在后续阶段再扩展 `MockDocument.content` 字段。

验收标准：

1. 文档页打开时编辑器内有可编辑的初始内容。
2. 初始内容主题应贴合当前企业文档协同项目。
3. 初始内容不应继续以不可编辑 HTML 或静态 JSX 形式写死在 `EditorShell` 中。

### 7.4 类型与数据模型

`packages/editor` 需要定义项目级编辑器内容类型，避免业务层直接绑定 BlockNote 类型。

推荐定义：

```ts
export type EditorBlock = {
  id?: string;
  type: string;
  props?: Record<string, unknown>;
  content?: unknown;
  children?: EditorBlock[];
};

export type EditorDocumentContent = EditorBlock[];
```

说明：

1. 本阶段类型可以与 BlockNote block 结构保持接近。
2. 业务层优先使用 `EditorDocumentContent`。
3. BlockNote 原始类型应尽量限制在 `packages/editor` 内部。

验收标准：

1. `packages/editor` 对外导出项目级编辑器类型。
2. `apps/web` 不需要直接 import `@blocknote/core` 类型。
3. 后续数据库保存可以基于该类型继续演进。

### 7.5 内容变化回调

编辑器需要提供内容变化回调，为后续保存、自动保存、RAG 索引和版本历史预留入口。

本阶段只要求在编辑器内容变化时能够向上层返回当前内容，不要求持久化。

验收标准：

1. `DocumentEditor` 支持 `onChange`。
2. `onChange` 返回项目级 `EditorDocumentContent`。
3. 当前阶段可以只在控制台或组件状态中消费该回调，不要求发起 API 请求。

### 7.6 样式与页面融合

BlockNote 编辑器需要融入当前 `ai-doc-platform` 的文档页视觉结构。

要求：

1. 保留当前白色主卡片和浅灰边框风格。
2. 编辑器不应出现独立 demo 页面中的大面积装饰背景。
3. 编辑器内容宽度需要适合当前中间主区域。
4. BlockNote 默认样式可以使用，但需要避免和 Tailwind 全局样式明显冲突。
5. 文档状态面板需要与编辑器标题区域融合，不再要求文档页保留 360px 右侧 AI Assistant 面板。

验收标准：

1. 文档页整体仍然像工作台页面，而不是 BlockNote demo 页面。
2. 编辑器不遮挡 PresenceBar、Sidebar、Topbar 或文档状态面板。
3. 桌面端布局稳定。

## 8. AI 接入边界

BlockNote 已经提供 AI Extension，但本阶段不要求完整接入真实 AI 调用。

本阶段对 AI 的要求是明确边界：

1. `packages/editor` 可以预留 AI 菜单挂载点。
2. `packages/editor` 不负责定义业务 AI 动作。
3. `packages/editor` 不负责 RAG 上下文构建。
4. `packages/editor` 不负责模型 provider 和 API key 管理。
5. `packages/ai-workspace` 后续负责 AI action、prompt、上下文和引用策略。
6. 文档页本阶段不要求渲染 `apps/web/src/components/ai/ai-assistant-panel.tsx`，AI Assistant 面板可以保留在代码中供后续页面或阶段复用。

后续关系应为：

```text
packages/ai-workspace
  定义 AI 动作、提示词、上下文策略

packages/editor
  将 AI 动作映射到 BlockNote 内嵌 AI 菜单

apps/web
  决定文档状态面板、编辑器内 AI 入口和后续 AI 工作流如何组合
```

验收标准：

1. 本阶段不在 `packages/editor` 中硬编码复杂 AI prompt。
2. 本阶段不在 `apps/web` 中直接实现 BlockNote AI 菜单。
3. 后续接入 AI 时有明确扩展点。

## 9. 协同接入边界

BlockNote 支持通过 Yjs provider 接入协同编辑，但本阶段不要求实现真实协同。

本阶段对协同的要求是明确边界：

1. `packages/editor` 未来负责接收 BlockNote collaboration option。
2. `packages/editor` 不负责创建 Yjs provider。
3. `packages/editor` 不负责管理协同房间、用户身份和权限。
4. `packages/collaboration` 后续负责 Yjs provider、roomId、presence 和同步状态。
5. `apps/collab` 后续负责 Hocuspocus 或其他 WebSocket 服务。
6. `PresenceBar` 本阶段继续使用 mock 数据。

后续关系应为：

```text
packages/collaboration
  创建 provider、房间和用户协同状态

packages/editor
  消费 collaboration 配置并传给 BlockNote

apps/web
  根据当前文档和当前用户组合协同配置
```

验收标准：

1. 本阶段不引入真实 Yjs provider。
2. 本阶段不修改 `apps/collab` 占位服务。
3. 编辑器组件类型设计应能在后续接收协同配置。

## 10. 工程需求

### 10.1 依赖管理

BlockNote 相关依赖应优先放在 `packages/editor` 中。

推荐依赖：

```text
@blocknote/core
@blocknote/react
@blocknote/mantine
```

如本阶段不接入真实 AI，则暂不必须引入：

```text
@blocknote/xl-ai
ai
```

说明：

1. 当前 `apps/web` 使用 React 18 和 Next.js 14。
2. demo 项目使用 React 19、TypeScript 6、Vite 8，不能直接照搬依赖版本。
3. 引入 BlockNote 时需要选择与当前 React 18 / TypeScript 5.4 兼容的版本。
4. 不应为了迁移 demo 而升级整个项目到 React 19 或 TypeScript 6。

验收标准：

1. BlockNote 依赖不会散落到多个业务包。
2. 不升级当前项目主框架版本，除非后续有单独技术决策。
3. `pnpm install` 后依赖关系清晰。

### 10.2 Next.js 客户端边界

BlockNote 是浏览器端编辑器能力，需要处理 Next.js App Router 中的客户端组件边界。

要求：

1. 使用 BlockNote 的组件文件需要声明 `"use client"`。
2. 如遇 SSR 兼容问题，可以在 `apps/web` 或 `packages/editor` 的集成层使用 dynamic import。
3. 不应把整个文档详情页都无必要地改成复杂客户端状态页面。

验收标准：

1. 文档页可以正常服务端渲染页面外壳。
2. 编辑器可以在客户端正常挂载。
3. 构建和类型检查不应出现浏览器 API 在服务端执行的问题。

### 10.3 对外导出

`packages/editor/src/index.ts` 应作为唯一稳定出口。

推荐导出：

```ts
export { DocumentEditor } from "./blocknote/document-editor";
export type {
  DocumentEditorProps,
  EditorBlock,
  EditorDocumentContent,
  DocumentEditorHandle,
} from "./types";
```

验收标准：

1. `apps/web` 只从 `@ai-doc-platform/editor` 导入编辑器能力。
2. `apps/web` 不从 `packages/editor/src/blocknote/*` 深路径导入。
3. 后续内部目录调整不影响业务页面。

## 11. 非功能需求

### 11.1 可维护性

编辑器底层依赖需要集中封装，避免业务页面直接依赖 BlockNote API。

### 11.2 可替换性

虽然当前选择 BlockNote，但项目级类型和组件 API 不应完全暴露 BlockNote 实现细节。后续如果需要升级 BlockNote 或调整编辑器方案，应优先修改 `packages/editor`，而不是大面积修改 `apps/web`。

### 11.3 可扩展性

本阶段完成后，应能继续扩展：

1. 文档保存和自动保存。
2. AI 菜单和 AI Workspace 动作联动。
3. RAG 检索结果插入引用。
4. Yjs 多人协同。
5. 评论和批注。
6. 版本历史。

### 11.4 视觉一致性

编辑器需要服务于当前工作台页面，不应把 demo 的视觉样式整体迁移进来。

## 13. 任务拆分

### 13.1 调整 editor 包定位

1. 更新 `packages/editor` 的入口文件。
2. 创建 `src/blocknote` 目录。
3. 创建 `src/types.ts`。
4. 创建 `src/commands.ts`。
5. 明确 `packages/editor` 是 BlockNote Adapter，而不是自研编辑器内核。

### 13.2 接入 BlockNote

1. 安装与当前项目兼容的 BlockNote 依赖。
2. 创建 `DocumentEditor` 客户端组件。
3. 配置 BlockNote 默认样式。
4. 配置默认中文或基础字典策略。
5. 配置默认初始内容。

### 13.3 替换静态编辑区

3. 将 `DocumentEditor` 挂载到主编辑区域。

1. 文档数据模型：在 mock 数据中加入 `content` 字段，并逐步迁移到数据库。

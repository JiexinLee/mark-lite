# 04. ai-workspace 需求文档

## 1. 背景

在 `01. 搭建项目` 阶段中，项目已经创建了 `packages/ai-workspace` 领域包，用于承接 AI 助手和文档内 AI 交互能力。

在 `02. 书写静态页面` 阶段中，`apps/web` 已经创建了设置页，并在设置页中预留了「AI 配置」区域。当前该区域仍然是占位状态，只展示“即将接入”。

在 `03. editor 需求说明书` 阶段中，`packages/editor` 被定位为 BlockNote Adapter。BlockNote 已经提供编辑器内 AI Extension、AI 菜单、选区处理和文档操作协议，因此 `packages/ai-workspace` 不应重复实现 BlockNote 的编辑器内 AI 执行能力。

`packages/ai-workspace` 应定位为项目级 AI 业务层：定义企业文档工作台中的 AI 动作、模型配置、prompt 构建、文档上下文、请求协议和扩展策略。BlockNote AI 只是这些 AI 能力的一个执行入口。

当前 `packages/ai-workspace` 仍然是占位包，仅导出 `aiWorkspacePlaceholder`。本阶段目标是将其从占位包演进为可被 `apps/web`、`packages/editor` 和后续服务端 AI 调用共同消费的领域包。

## 2. 目标

### 2.1 业务目标

建立文档工作台的 AI 能力基础，使项目能够明确回答以下问题：

1. 系统支持哪些 AI 动作。
2. 每个 AI 动作属于哪个业务分类。
3. 每个 AI 动作需要什么文档上下文。
4. 每个 AI 动作如何构建 prompt。
5. AI 模型配置应包含哪些字段。
6. 设置页的「AI 配置」区域应如何承接模型配置。
7. 后续 editor 内嵌 AI 菜单、设置页、RAG 引用和服务端模型调用如何共享同一套 AI 业务定义。

### 2.2 技术目标

1. 将 `packages/ai-workspace` 从占位包改造成 AI Workspace 领域包。
2. 在 `packages/ai-workspace` 中定义 AI action registry。
3. 在 `packages/ai-workspace` 中定义 AI 模型配置结构和默认值。
4. 在 `packages/ai-workspace` 中定义 provider 描述信息。
5. 在 `packages/ai-workspace` 中定义文档上下文结构。
6. 在 `packages/ai-workspace` 中定义 prompt builder。
7. 在 `packages/ai-workspace` 中定义前后端 AI 请求协议。
8. 在 `packages/ai-workspace` 中预留权限、审计、额度、隐私脱敏等扩展口，但不实现复杂策略。
9. 在 `apps/web` 设置页中让「AI 配置」区域能够展示第一版 AI 配置结构。
10. 保持当前项目不接入真实 AI 模型调用，不保存真实 API key。

## 3. 范围

### 3.1 本阶段包含

1. 在 `packages/ai-workspace` 中创建领域目录结构。
2. 定义 AI 动作类型和默认动作列表。
3. 提供写作类动作：改写、润色、扩写。
4. 提供翻译类动作：翻译成英文。
5. 提供文档类动作：总结当前文档。
6. 提供研究类动作：从知识库查找引用。
7. 定义 AI provider 类型和 provider 描述。
8. 定义 AI 模型配置类型。
9. 定义 AI Workspace 设置类型。
10. 提供默认 AI Workspace 设置。
11. 定义文档上下文、选区上下文和 RAG 片段上下文类型。
12. 提供基础 prompt 构建函数。
13. 定义 AI Workspace 请求类型。
14. 定义策略扩展类型，但不实现复杂策略逻辑。
15. 在设置页的「AI 配置」区域展示 provider、model、baseUrl、temperature、maxTokens、enabledActions 等配置项。
16. 保持配置为前端 mock 或本地状态，不接入真实后端保存。

### 3.2 本阶段不包含

1. 不接入 OpenAI、DeepSeek、Claude 或其他真实模型 API。
2. 不实现服务端 AI route。
3. 不实现真实 API key 保存。
4. 不在浏览器长期保存真实 API key。
5. 不实现 BlockNote AI Extension 的完整接入。
6. 不实现 AI 结果写回编辑器。
7. 不实现 RAG 真实检索。
8. 不实现权限系统。
9. 不实现审计日志落库。
10. 不实现用户额度和 token 计费。
11. 不实现隐私脱敏逻辑。
12. 不实现 prompt 版本管理后台。
13. 不实现 provider 连接测试。
14. 不实现复杂模型能力矩阵。

本阶段只负责建立 AI Workspace 领域定义和设置页配置承接能力，为后续真实 AI 调用和编辑器内 AI 接入打基础。

## 4. 用户与使用场景

### 4.1 目标用户

本阶段主要面向项目开发者、课程学习者和后续模块维护者。

开发者需要通过本阶段明确：

1. BlockNote AI 是编辑器内 AI 执行入口，不是项目全部 AI 业务层。
2. `packages/ai-workspace` 是 AI 动作、配置、prompt 和请求协议的领域中心。
3. `apps/web` 设置页只负责展示和编辑 AI 配置 UI。
4. `packages/editor` 后续只消费 AI 动作定义，并将其映射到 BlockNote AI 菜单。
5. 真实模型 provider 和 API key 管理应放到后续服务端阶段。

### 4.2 核心场景

开发者启动项目后，可以完成以下认知和操作：

1. 在 `packages/ai-workspace` 中查看平台支持的 AI 动作列表。
2. 在 `packages/ai-workspace` 中查看默认模型配置。
3. 在 `packages/ai-workspace` 中使用 action + context 构建 prompt。
4. 在设置页中看到「AI 配置」从占位区域升级为可展示配置的区域。
5. 明确后续 editor、RAG、server API 应如何消费 `ai-workspace`。

## 5. 架构定位

### 5.1 包职责

推荐边界如下：

```text
apps/web
  负责设置页 UI、页面组合、用户输入和 mock 状态展示

packages/ai-workspace
  负责 AI 动作、模型配置、provider 定义、prompt 构建、上下文结构、请求协议和策略扩展口

packages/editor
  负责将 AI 动作映射到 BlockNote 编辑器内 AI 菜单和文档操作能力

packages/rag
  后续负责真实知识检索、chunk、embedding、rerank 和 citation 数据来源

server / api route
  后续负责读取配置、创建模型 client、调用模型并返回结果
```

### 5.2 与 BlockNote AI 的关系

BlockNote AI 负责：

1. 编辑器内 AI 菜单。
2. 选区和光标附近内容处理。
3. AI 结果应用到文档 block。
4. 接受、撤销、重试等编辑器内交互。

`packages/ai-workspace` 负责：

1. 定义 AI 动作。
2. 定义动作分类、标题、描述和 prompt。
3. 定义动作需要的上下文。
4. 定义模型配置。
5. 定义请求协议。
6. 定义后续权限、审计、额度、脱敏等扩展口。

说明：

1. `ai-workspace` 不直接渲染 BlockNote AI 菜单。
2. `ai-workspace` 不直接操作编辑器 selection。
3. `ai-workspace` 不直接调用 `editor.insertBlocks`。
4. `ai-workspace` 不直接创建模型 client。
5. `ai-workspace` 不保存真实 API key。

### 5.3 与设置页的关系

当前 `apps/web/src/app/(workspace)/settings/page.tsx` 中已经存在「AI 配置」区域，但仍是占位。

本阶段设置页应从 `packages/ai-workspace` 获取：

1. 默认 AI Workspace 设置。
2. Provider 选项。
3. 默认模型名称。
4. 默认温度和 token 参数。
5. 可启用的 AI action 列表。

设置页本阶段只展示和编辑 mock 配置，不发起真实 API 请求。

`packages/ai-workspace` 推荐调整为：

```text
packages/ai-workspace
├── package.json
├── tsconfig.json
└── src
    ├── actions
    │   ├── writing.ts
    │   ├── translation.ts
    │   ├── research.ts

    ├── config

    │   └── defaults.ts
    ├── context

    ├── contracts
    │   └── request.ts
    ├── policies
    │   └── types.ts
    ├── types.ts
    └── index.ts
```

说明：

1. `actions` 存放 AI 动作定义。
2. `config` 存放 provider、模型配置和默认设置。
3. `context` 存放文档、选区、RAG 上下文类型。
4. `prompts` 存放 prompt 构建函数。
5. `contracts` 存放前后端请求协议。
6. `policies` 只存放扩展类型，不实现复杂策略。
7. `index.ts` 作为对外唯一出口。

## 7. 产品需求

### 7.1 AI 动作定义

`packages/ai-workspace` 需要定义平台第一版 AI 动作。

推荐动作：

| key | 分类 | 名称 | 说明 |
| :--- | :--- | :--- | :--- |
| `rewrite` | `writing` | 改写 | 在不改变原意的前提下重写当前内容 |
| `polish` | `writing` | 润色 | 优化语言质量、语气和可读性 |
| `expand` | `writing` | 扩写 | 基于当前内容补充细节和示例 |
| `translate-to-en` | `translation` | 翻译成英文 | 将当前内容翻译成自然准确的英文 |
| `summarize-document` | `document` | 总结当前文档 | 基于文档上下文生成摘要 |
| `find-citations` | `research` | 从知识库查找引用 | 基于当前文档或选区构造检索意图 |

推荐类型：

```ts
export type AIWorkspaceActionCategory =
  | "writing"
  | "translation"
  | "document"
  | "research";

export type AIWorkspaceAction = {
  key: string;
  title: string;
  description: string;
  category: AIWorkspaceActionCategory;
  prompt: string;
  requiresSelection?: boolean;
  requiresRag?: boolean;
};
```

验收标准：

1. `packages/ai-workspace` 对外导出默认动作列表。
2. 每个动作有稳定 `key`。
3. 每个动作有标题、描述、分类和 prompt。
4. 动作定义不依赖 React。
5. 动作定义不依赖 BlockNote。

### 7.2 AI 模型配置

`packages/ai-workspace` 需要定义 AI 模型配置结构。

推荐类型：

```ts
export type AIProvider = "openai" | "deepseek" | "anthropic" | "custom";

export type AIModelConfig = {
  provider: AIProvider;
  model: string;
  baseUrl?: string;
  apiKeyRef?: string;
  temperature?: number;
  maxTokens?: number;
};
```

说明：

1. `apiKeyRef` 表示服务端密钥引用，不表示前端明文 API key。
2. 本阶段不保存真实 API key。
3. `custom` provider 用于 OpenAI-compatible 服务。

验收标准：

1. `packages/ai-workspace` 对外导出 `AIProvider` 和 `AIModelConfig`。
2. 不在包内写死真实 API key。
3. 不要求本阶段创建模型 client。

### 7.3 Provider 描述

`packages/ai-workspace` 需要提供 provider 描述信息，供设置页展示。

推荐类型：

```ts
export type AIProviderDefinition = {
  provider: AIProvider;
  label: string;
  description: string;
  defaultBaseUrl?: string;
  defaultModel: string;
  supportsCustomBaseUrl?: boolean;
};
```

推荐第一版 provider：

1. OpenAI。
2. DeepSeek。
3. Anthropic。
4. Custom OpenAI-compatible。

验收标准：

1. 设置页可以从 `packages/ai-workspace` 获取 provider 列表。
2. provider 描述不包含真实密钥。
3. provider 描述只表达展示和默认配置，不创建实际 SDK client。

### 7.4 AI Workspace 设置

`packages/ai-workspace` 需要定义 workspace 级 AI 设置。

推荐类型：

```ts
export type AIWorkspaceSettings = {
  defaultModel: AIModelConfig;
  enabledActions: string[];
  citationRequired: boolean;
};
```

推荐默认值：

1. 默认 provider：`deepseek` 或 `custom`。
2. 默认 model：`deepseek-chat` 或一个可替换的占位模型名。
3. 默认启用第一版所有动作。
4. 默认 `citationRequired` 为 `false`。

验收标准：

1. `packages/ai-workspace` 导出默认设置。
2. 默认设置可以被 `apps/web` 设置页展示。
3. 默认设置不触发任何真实网络请求。

### 7.5 文档上下文

`packages/ai-workspace` 需要定义 AI 请求使用的文档上下文。

推荐类型：

```ts
export type AIWorkspaceDocumentContext = {
  documentId: string;
  title: string;
  selectedText?: string;
  surroundingText?: string;
  documentText?: string;
  ragSnippets?: AIWorkspaceRagSnippet[];
};

export type AIWorkspaceRagSnippet = {
  id: string;
  title: string;
  source: string;
  excerpt: string;
  score?: number;
  metadata?: Record<string, unknown>;
};
```

说明：

1. 本阶段只定义结构，不要求从 BlockNote blocks 自动抽取文本。
2. 后续 `packages/editor` 可以将选区内容转换成 `selectedText`。
3. 后续 `packages/rag` 可以提供 `ragSnippets`。

验收标准：

1. 上下文类型不依赖 BlockNote。
2. 上下文类型可以同时服务 editor 和 RAG。
3. 上下文结构可以用于 prompt 构建。

### 7.6 Prompt 构建

`packages/ai-workspace` 需要提供基础 prompt 构建函数。

推荐输入：

```ts
export type BuildAIWorkspacePromptInput = {
  actionKey: string;
  context: AIWorkspaceDocumentContext;
};
```

推荐输出：

```ts
export type AIWorkspacePrompt = {
  system: string;
  user: string;
};
```

要求：

1. 根据 actionKey 找到对应动作。
2. 将文档标题、选区、全文摘要或 RAG 片段拼入 user prompt。
3. 不在 prompt builder 中调用模型。
4. 不在 prompt builder 中访问浏览器或服务端环境变量。

验收标准：

1. 可以基于 action + context 生成 prompt。
2. prompt builder 是纯函数。
3. prompt builder 不依赖 React、Next.js、BlockNote。

### 7.7 AI 请求协议

`packages/ai-workspace` 需要定义 AI Workspace 请求结构，为后续 API route 和 editor AI 菜单复用。

推荐类型：

```ts
export type AIWorkspaceRequest = {
  actionKey: string;
  documentId: string;
  context: AIWorkspaceDocumentContext;
  settings: AIWorkspaceSettings;
};
```

说明：

1. 本阶段只定义请求结构。
2. 不实现请求发送。
3. 不实现响应解析。
4. 后续 API route 可以直接复用该类型。

验收标准：

1. `packages/ai-workspace` 对外导出请求类型。
2. 请求类型不绑定具体模型 provider SDK。
3. 请求类型可以被 editor 和 web 共同使用。

### 7.8 策略扩展口

本阶段不实现复杂策略，但需要预留扩展口。

推荐类型：

```ts
export type AIWorkspacePolicyHooks = {
  canRunAction?: (input: AIWorkspacePolicyInput) => boolean;
  sanitizeContext?: (
    context: AIWorkspaceDocumentContext,
  ) => AIWorkspaceDocumentContext;
  createAuditEvent?: (
    input: AIWorkspaceAuditInput,
  ) => AIWorkspaceAuditEvent;
};
```

扩展口用于后续承接：

1. 权限控制。
2. 审计日志。
3. 用户额度。
4. token 成本。
5. 隐私脱敏。
6. prompt 版本管理。
7. provider 能力矩阵。

验收标准：

1. 本阶段只导出类型，不实现复杂业务逻辑。
2. 默认策略不阻止任何 action。
3. 后续可以在不推翻包结构的情况下增加策略实现。

AI 配置区域至少展示：

1. Provider。
2. Model。
3. Base URL。
4. Temperature。
5. Max Tokens。
6. Enabled Actions。

本阶段交互可以保持轻量：

3. 可以不提供真实保存 API。
4. 可以不提供连接测试。
5. 不展示真实 API key 输入框，或仅展示 API key 将由服务端配置的提示。

验收标准：

1. 设置页不再只显示「AI 配置 / 即将接入」。
2. 设置页从 `packages/ai-workspace` 获取默认设置和 provider/action 列表。

4. 设置页不发起真实 AI 请求。

## 9. 工程需求

### 9.1 依赖管理

`packages/ai-workspace` 本阶段应保持轻量，优先只使用 TypeScript。

要求：

1. 不引入 AI SDK。
2. 不引入 OpenAI SDK。
3. 不引入 LangChain。
4. 不引入 LlamaIndex。
5. 不引入 React。
6. 不引入 BlockNote。

验收标准：

1. `packages/ai-workspace` 可以作为纯 TypeScript 领域包使用。
2. `apps/web` 可以从该包导入类型、默认配置和动作列表。
3. `packages/editor` 后续可以从该包导入动作列表，而不会引入 web 依赖。

3. 不破坏现有 `packages/editor` 类型检查。

## 10. 非功能需求

### 10.1 可维护性

AI action、prompt、settings 和 request contract 需要集中在 `packages/ai-workspace`，避免散落在设置页、编辑器组件或后续 API route 中。

### 10.2 可扩展性

本阶段只实现最小可用业务定义，但目录和类型需要能扩展：

1. 权限控制。
2. 审计日志。
3. 用户额度。
4. 成本统计。
5. 隐私脱敏。
6. prompt 版本。
7. provider 能力矩阵。
8. 多模型路由。

### 10.3 安全性

本阶段不处理真实密钥，但文档和类型设计需要明确：

1. 不建议前端保存明文 API key。
2. `apiKeyRef` 只是服务端密钥引用。
3. 真实模型调用应在后续服务端实现。
4. 设置页不应误导用户把真实密钥存入 localStorage。

### 10.4 与现有 UI 一致性

设置页 AI 配置区域应延续当前工作台视觉风格：

1. 白色卡片。
2. 浅灰边框。
3. 克制的信息密度。
4. 不加入大段解释型文案。
5. 不把设置页做成复杂控制台。

# 02. 书写静态页面

## 1. 背景

在 `01. 搭建项目` 阶段中，项目已经完成 monorepo 工程底座搭建，包含 Next.js 主应用、协同服务占位应用，以及编辑器、AI Workspace、RAG、协同、权限、数据库、共享能力等领域包。

当前项目的核心定位不是传统企业后台，也不是普通官网，而是面向企业知识沉淀、协同编辑、AI 辅助创作和知识检索的文档工作空间。

因此，页面搭建不应优先围绕用户管理、权限管理、CRUD 表格或 Dashboard 图表展开，而应优先建立以下核心界面结构：

1. Workspace 工作台布局。
2. 左侧文档树和空间导航。
3. 中间文档编辑器区域。
4. 右侧 AI Assistant 面板。
5. 协同在线状态和同步状态区域。
6. RAG 知识检索工作台。

本阶段只完成静态页面，不接入真实业务数据、不实现完整编辑器、不实现协同服务、不实现 RAG 和 AI 调用。页面需要为后续 BlockNote、Yjs / Hocuspocus、RAG、AI Assistant 等模块预留稳定的 UI 区域和工程边界。

整体产品风格参考 Notion、飞书文档、语雀、Outline、Cursor、ChatGPT Canvas 等文档工作空间产品，强调清晰、克制、高密度、可持续扩展的工作台体验。

## 2. 目标

### 2.1 业务目标

搭建企业级协同知识库的静态页面骨架，使项目从基础启动页进入可感知的文档工作空间形态。

用户进入系统后，应能直观看到：

1. 这是一个文档工作空间。
2. 这是一个围绕文档编辑展开的产品。
3. AI、协同、RAG 检索是工作空间中的核心能力，而不是附属功能。
4. 后续核心模块有明确的页面承载区域。

### 2.2 技术目标

1. 基于 Next.js App Router 创建 Workspace 路由结构。
2. 使用 Tailwind CSS 建立基础样式体系。
3. 使用 shadcn/ui 作为基础 UI 组件来源。
4. 不引入 Ant Design、Material UI、Element Plus 等额外组件库。
5. 创建登录页面，为后续多浏览器、多用户协同测试预留入口。
6. 抽离可复用的 Workspace Layout。
7. 抽离左侧导航、顶部栏、编辑器占位、AI 面板、协同状态、RAG 工作台等组件。
8. 保持页面为静态实现，使用 mock data 表达产品结构。
9. 保证页面结构能承接后续领域包能力集成。

## 4. 用户与使用场景

### 4.1 目标用户

本阶段页面主要面向项目开发者、课程学习者和后续模块维护者。

开发者需要通过静态页面明确：

1. 文档工作空间的整体布局方式。
2. 文档编辑页是系统核心页面。
3. AI Assistant 应该在文档上下文中工作。
4. 协同状态应该和文档编辑体验结合。
5. RAG 查询既可以独立工作，也可以在文档页中被引用。
6. 后续领域包应该集成到哪些 UI 区域。

### 4.2 核心场景

开发者启动 `apps/web` 后，可以访问以下页面：

1. 登录页：使用硬编码账号密码进入工作台，为后续协同测试准备用户身份入口。
2. Workspace 首页：查看工作空间概览、最近文档和核心入口。
3. 文档编辑页：查看文档树、编辑器占位、AI 面板和协同状态。
4. RAG 查询页：查看知识检索输入框、检索结果、chunk、source、score 等静态信息。
5. 设置页占位：为后续工作空间配置预留入口。

## 5. 页面规划

### 5.1 推荐路由结构

基于当前 Next.js App Router 项目结构，页面应放在 `apps/web/src/app` 下。

推荐结构如下：

```text
apps/web/src/app
├── layout.tsx
├── page.tsx
├── login
│   └── page.tsx
└── (workspace)
    ├── layout.tsx
    ├── page.tsx
    ├── docs
    │   └── [id]
    │       └── page.tsx
    ├── search
    │   └── page.tsx
    └── settings
        └── page.tsx
```

说明：

1. `app/page.tsx` 可以作为根路径入口，也可以跳转到 `/login` 或复用 Workspace 首页。
2. `app/login/page.tsx` 承载静态登录页面。
3. `(workspace)/layout.tsx` 承载工作台通用布局。
4. `(workspace)/docs/[id]/page.tsx` 是后续项目最核心页面。
5. `(workspace)/search/page.tsx` 独立承载 RAG 查询工作台。
6. `(workspace)/settings/page.tsx` 本阶段仅做占位。

### 5.2 推荐组件结构

```text
apps/web/src/components
├── auth
│   └── login-form.tsx
├── layout
│   ├── workspace-layout.tsx
│   ├── workspace-sidebar.tsx
│   └── workspace-topbar.tsx
├── editor
│   └── editor-shell.tsx
├── ai
│   └── ai-assistant-panel.tsx
├── collaboration
│   └── presence-bar.tsx
├── rag
│   └── rag-search-workbench.tsx
└── ui
    └── ...
```

说明：

1. `components/auth` 存放登录相关 UI，本阶段只实现硬编码登录表单。
2. `components/layout` 存放页面骨架相关组件。
3. `components/editor` 存放编辑器相关 UI，占位对应 `packages/editor`。
4. `components/ai` 存放 AI Assistant 相关 UI，占位对应 `packages/ai-workspace`。
5. `components/collaboration` 存放协同状态相关 UI，占位对应 `packages/collaboration` 和 `apps/collab`。
6. `components/rag` 存放知识检索相关 UI，占位对应 `packages/rag` 和 `packages/database`。
7. `components/ui` 存放 shadcn/ui 组件。

## 6. 产品需求

### 6.1 登录页

登录页用于为后续文档协同测试提供基础用户身份入口。由于后续需要在不同浏览器中登录不同用户来测试多人协同，本阶段需要先具备一个简单的登录页面和硬编码登录流程。

推荐路径：

```text
/login
```

页面内容包含：

1. 产品名称：企业级协同知识库。
2. 登录表单：
   - 用户名
   - 密码
   - 登录按钮
3. 测试账号提示：
   - 用户名：`admin`
   - 密码：`123456`
4. 登录失败提示占位。
5. 进入工作台后的用户身份展示占位。

本阶段推荐使用以下硬编码账号：

```text
username: admin
password: 123456
```

后续为了测试多人协同，可以扩展更多硬编码测试用户，例如：

```text
username: editor
password: 123456
```

本阶段不要求实现真实认证，可以使用组件内部状态或浏览器本地状态模拟登录成功。登录成功后进入 Workspace 首页。

验收标准：

1. `/login` 页面可以访问。
2. 页面包含用户名和密码输入框。
3. 使用 `admin / 123456` 可以模拟登录成功。
4. 登录成功后可以进入 Workspace 首页。
5. 错误账号密码需要有静态错误提示。
6. 登录状态不需要和后端服务交互。
7. 不需要实现注册、找回密码、验证码和第三方登录。

### 6.2 Workspace Layout

Workspace Layout 是所有工作台页面的基础布局。

推荐布局：

```text
┌──────────────────────────────────────────────┐
│ Topbar                                       │
├──────────────┬───────────────────────────────┤
│ Sidebar      │ Page Content                  │
│              │                               │
│              │                               │
│              │                               │
└──────────────┴───────────────────────────────┘
```

Layout 需要包含：

1. 左侧 Sidebar。
2. 顶部 Topbar。
3. 主内容区。
4. 当前空间名称。
5. 当前登录用户展示。
6. 页面级操作入口。

验收标准：

1. Workspace 页面共享同一套布局。
2. Sidebar 和 Topbar 视觉稳定，不随页面内容跳动。
3. 主内容区可以承载首页、文档页、搜索页和设置页。
4. Topbar 需要能展示当前模拟登录用户。
5. 布局风格接近文档工作空间，而不是传统后台管理系统。

### 6.3 Workspace Sidebar

Sidebar 用于展示工作空间导航和文档树。

Sidebar 需要包含：

1. 产品名称或工作空间名称。
2. Workspace 首页入口。
3. 文档入口。
4. RAG Search 入口。
5. Settings 入口。
6. 静态文档树。
7. 最近访问或收藏文档。

文档树示例：

```text
产品知识库
├── 项目简介
├── 协同编辑方案
├── RAG 检索设计
└── AI 助手交互
```

验收标准：

1. Sidebar 能体现文档工作空间的层级关系。
2. 文档树为静态数据即可。
3. 当前页面或当前文档需要有明显选中态。
4. 不需要实现真实展开收起逻辑。

### 6.4 Workspace 首页

Workspace 首页用于建立工作台感，展示最近文档和核心功能入口。

推荐路径：

```text
/(workspace)
```

页面内容包含：

1. 当前工作空间标题。
2. 最近编辑文档。
3. 快速入口：
   - 新建文档占位
   - 打开 RAG Search
   - 查看 AI Workspace
4. 静态活动记录。
5. 协同状态概览。

验收标准：

1. 页面能证明系统已经从启动页进入工作台形态。
2. 页面不应设计成营销落地页。
3. 页面不应设计成传统 BI Dashboard。
4. 页面视觉应简洁、克制，适合长期使用。

### 6.5 文档编辑页

文档编辑页是本项目最核心页面。

推荐路径：

```text
/(workspace)/docs/[id]
```

推荐布局：

```text
┌────────────────────────────────────────────────────┐
│ Topbar                                             │
├──────────────┬───────────────────────┬─────────────┤
│ Sidebar      │ Editor                │ AI Panel    │
│              │                       │             │
│ Document     │ Title                 │ Ask / Edit  │
│ Tree         │ Blocks                │ Suggestions │
│              │                       │ Context     │
└──────────────┴───────────────────────┴─────────────┘
```

页面内容包含：

1. 文档标题。
2. 文档元信息：
   - 更新时间
   - 作者
   - 同步状态
3. 编辑器静态占位区域。
4. 文档正文静态 blocks。
5. 协同在线状态。
6. 右侧 AI Assistant 面板。
7. AI 操作入口占位：
   - 总结
   - 润色
   - 改写
   - 翻译
   - 基于选区提问
8. 引用知识库入口占位。

验收标准：

1. 页面应明显体现“文档编辑”是核心。
2. 编辑器区域应是视觉主区域。
3. AI Assistant 应作为文档上下文能力存在，而不是独立聊天页。
4. 协同状态需要在页面中有可见位置。
5. 页面可以使用静态 blocks 模拟编辑器内容。
6. 不需要实现真实输入、保存和协同。

### 6.6 Editor Shell

Editor Shell 用于模拟后续 BlockNote 编辑器的承载区域。

静态内容可以包含：

1. 文档标题。
2. 段落 block。
3. 标题 block。
4. 引用 block。
5. 列表 block。
6. 代码或命令占位。
7. Slash command 提示占位。

验收标准：

1. 编辑器区域应看起来像可编辑文档，而不是普通详情页。
2. block 之间应有清晰层次。
3. 需要为后续接入 BlockNote 预留组件边界。
4. 不需要实现真实编辑行为。

### 6.7 AI Assistant Panel

AI Assistant Panel 用于模拟后续文档内 AI 能力。

面板内容包含：

1. 当前上下文提示。
2. AI 快捷操作：
   - 总结当前文档
   - 改写选中内容
   - 生成行动项
   - 从知识库查找引用
3. 静态对话消息。
4. 输入框占位。
5. 引用来源占位。

验收标准：

1. AI Panel 应位于文档页右侧。
2. 面板宽度应稳定，不挤压到编辑器不可用。
3. AI 能力应围绕文档上下文表达。
4. 不需要真实调用模型 API。

### 6.8 Collaboration Presence

协同状态组件用于模拟多人在线和同步状态。

内容包含：

1. 在线成员头像或缩写。
2. 在线人数。
3. 当前同步状态。
4. 最近编辑提示。

状态示例：

```text
3 人在线
已同步
Lin 正在查看 RAG 检索设计
```

验收标准：

1. 文档页需要展示协同状态。
2. Workspace 首页可以展示协同概览。
3. 只使用静态数据。
4. 不需要 WebSocket 或真实 presence。

### 6.9 RAG 查询页

RAG 查询页用于后续承载企业知识检索、chunk 调试和 citation 展示。

推荐路径：

```text
/(workspace)/search
```

推荐布局：

```text
┌──────────────────────────────────────┐
│ Search Input                         │
├──────────────────────────────────────┤
│ Result List                          │
│ ├── chunk                            │
│ ├── source                           │
│ ├── score                            │
│ └── metadata                         │
└──────────────────────────────────────┘
```

页面内容包含：

1. 搜索输入框占位。
2. 查询建议。
3. 检索结果列表。
4. chunk 摘要。
5. source 文档。
6. score / rerank 静态信息。
7. metadata 静态信息。
8. citation 占位。

验收标准：

1. 页面应体现这是知识检索工作台，而不是普通站内搜索。
2. 检索结果需要展示 RAG 调试所需信息。
3. 不需要真实检索。
4. 不需要接入向量数据库。

### 6.10 Settings 占位页

Settings 页面本阶段只作为工作空间配置入口占位。

推荐路径：

```text
/(workspace)/settings
```

页面内容包含：

1. 工作空间基础信息占位。
2. 成员与权限占位。
3. AI 配置占位。
4. 知识库索引配置占位。

验收标准：

1. 页面可以从 Sidebar 进入。
2. 页面明确说明后续配置方向。
3. 不需要实现真实表单提交。

## 7. 视觉与交互要求

### 7.1 技术栈要求

本阶段样式和组件要求：

1. 使用 Tailwind CSS 组织样式。
2. 使用 shadcn/ui 作为基础组件来源。
3. 可以使用 shadcn/ui 的 Button、Card、Input、Badge、Avatar、Separator、ScrollArea 等基础组件。
4. 不安装 Ant Design、Material UI、Element Plus 等额外组件库。
5. 不引入复杂状态管理库。
6. 静态数据可以直接写在组件文件中，或抽离到本地常量文件。

### 7.2 视觉风格

整体风格参考 Notion 和飞书文档：

1. 背景干净。
2. 层级克制。
3. 留白适中。
4. 信息密度高于营销页。
5. 操作入口清晰。
6. 不使用夸张渐变和装饰图形。
7. 不做传统企业官网 Hero。
8. 不做传统后台 Dashboard 大屏。

### 7.3 布局要求

1. Sidebar 宽度稳定。
2. Topbar 高度稳定。
3. 文档编辑页采用三栏结构：
   - 左侧 Sidebar
   - 中间 Editor
   - 右侧 AI Panel
4. RAG 查询页采用主内容工作台结构。
5. 页面在常见桌面宽度下不应出现明显重叠。
6. 窄屏下可以简化为横向滚动或隐藏次要面板，本阶段不要求完整移动端体验。

## 8. 工程需求

### 8.1 Tailwind CSS

`apps/web` 需要接入 Tailwind CSS。

验收标准：

1. 存在 Tailwind 配置文件。
2. 存在全局样式入口。
3. `apps/web/src/app/layout.tsx` 正确引入全局样式。
4. 页面样式主要通过 Tailwind class 实现。

### 8.2 shadcn/ui

`apps/web` 需要具备 shadcn/ui 基础结构。

验收标准：

1. 存在 `components.json` 或等效 shadcn 配置。
2. 存在 `components/ui` 目录。
3. 页面使用 shadcn/ui 基础组件。
4. 不需要一次性安装大量组件，只添加当前静态页面需要的组件。

### 8.3 路由与组件边界

页面和组件需要保持清晰边界。

验收标准：

1. 路由文件只负责组织页面。
2. 复杂 UI 拆分到 `components`。
3. 登录、编辑器、AI、RAG、协同相关 UI 分别放到对应目录。
4. 不把所有页面 UI 都堆在单个 `page.tsx` 中。

### 8.4 Mock Data

本阶段允许使用静态 mock data。

推荐 mock 数据包括：

1. 测试用户。
2. workspace 信息。
3. 文档树。
4. 最近文档。
5. 在线成员。
6. AI 消息。
7. RAG 检索结果。

验收标准：

1. mock data 不依赖网络请求。
2. mock data 不包含真实密钥。
3. mock data 能表达后续真实数据结构的大致形态。
4. 测试账号只能用于本地静态演示，不应被描述为真实安全认证方案。

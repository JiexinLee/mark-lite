# 05. editor-ai 需求文档

## 1. 背景

在 `03. editor 需求说明书` 阶段中，项目已经将 `packages/editor` 定位为 BlockNote Adapter，并完成基础富文本编辑能力接入。当前文档编辑页已经可以使用 BlockNote 进行基础编辑。

在 `04. ai-workspace 需求文档` 阶段中，项目已经明确 `packages/ai-workspace` 的职责：定义 AI 动作、模型配置、prompt 构建、文档上下文、请求协议和策略扩展口。当前 `ai-workspace` 不负责渲染编辑器内 AI 菜单，也不直接调用模型。

之前的 `blocknote-demo` 项目已经验证过一套可参考的 BlockNote AI 集成方式，包括：

1. `BlockNoteAIMenu.tsx`：编辑器内 AI prompt 菜单。
2. `EditorAIMenuController.tsx`：AI 菜单弹层控制器。
3. `BlockNoteSlashMenu.tsx`：扩展 slash menu，加入 AI 助手入口。
4. `EditorFormattingToolbar.tsx`：选中文本后在格式化工具栏中加入 AI 按钮。
5. 服务端 `/api/chat`：使用 `@blocknote/xl-ai/server` 的 transport 对接真实模型。

当前 `ai-doc-platform` 已经具备 `editor` 和 `ai-workspace` 的基础，但还没有接入编辑器内 AI 交互层，也没有服务端真实模型调用链路。

本阶段目标是将 `ai-workspace` 定义的 AI 动作接入 BlockNote 编辑器内 AI 菜单，并使用用户在设置页「AI 配置」中录入的真实 API key 调用模型，使开发者和课程学习者可以在文档编辑页测试 AI 助手效果。

## 2. 目标

### 2.1 业务目标

让用户在文档编辑器中直接使用 AI 助手，对文档内容进行改写、润色、扩写、翻译和总结等操作。

用户进入 `/docs/[id]` 页面后，应能完成：

1. 选中文本后，通过格式化工具栏打开 AI 助手。
2. 输入 `/` 后，通过 slash menu 选择 AI 动作。
3. 选择 `ai-workspace` 中定义的 AI 动作。
4. 使用真实模型生成内容。
5. 在 BlockNote 编辑器内查看 AI 生成结果。
6. 接受或撤销 AI 对文档的修改。

### 2.2 技术目标

1. 在 `packages/editor` 中接入 `@blocknote/xl-ai`。
2. 在 `packages/editor` 中创建编辑器内 AI 菜单组件。
3. 在 `packages/editor` 中创建 AI 菜单控制器。
4. 在 `packages/editor` 中扩展 slash menu，加入 `ai-workspace` AI 动作入口。
5. 在 `packages/editor` 中扩展 formatting toolbar，加入 AI 按钮。
6. 将 `packages/ai-workspace` 的 action registry 映射为 BlockNote AI prompt。
7. 为 `DocumentEditor` 增加明确的 AI 扩展 props，避免把 AI 配置和 transport 逻辑硬编码在组件内部。
8. 在 `packages/editor` 中提供 BlockNote 文档上下文到 `AIWorkspaceDocumentContext` 的 adapter。
9. 在 `apps/web` 中创建固定的服务端 AI API route：`/api/ai/editor`。
10. 通过设置页「AI 配置」录入真实 API key，并在服务端 route 中使用该配置完成模型调用。
11. 使用真实模型 provider 完成 AI 请求。
12. 保证 API key 不进入前端 bundle、不写入源码、不提交到 Git，并避免在日志和 URL 中暴露。

## 3. 范围

### 3.1 本阶段包含

1. 安装并接入 `@blocknote/xl-ai`。
2. 安装并接入 AI SDK 所需依赖。
3. 在 `packages/editor` 中添加 AI 相关组件。
4. 在 `packages/editor` 中将 AI Extension 加入 BlockNote 初始化。
5. 在 `packages/editor` 中通过 `ai-workspace` action 生成 AI 菜单项。
6. 为 `DocumentEditor` 增加 AI 配置、transport 路径、上下文构建和错误处理相关 props。
7. 提供 BlockNote 文档上下文 adapter，用于构建 `AIWorkspaceDocumentContext`。
8. 支持选中文本后打开 AI 菜单。
9. 支持 slash menu 中选择 AI 动作。
10. 支持 AI 请求流式返回。
11. 支持 AI 结果应用到当前文档。
12. 支持接受或撤销 AI 修改。
13. 在 `apps/web` 中添加固定 route：`/api/ai/editor`。
14. 在服务端 route 中读取本次请求携带的 AI 配置和 API key。
15. 支持至少一个真实 provider，用于本地测试。
16. 更新设置页「AI 配置」说明，明确 API key 由用户在页面中录入。
17. 保持文档页当前布局：PresenceBar、文档状态面板、编辑器主体不被破坏。

### 3.2 本阶段不包含

1. 不恢复右侧 AI Assistant 面板。
2. 不实现独立 AI 聊天页。
3. 不实现 RAG 真实检索。
4. 不实现从 RAG 检索结果插入 citation。
5. 不实现协同编辑场景下的 AI 冲突处理。
6. 不实现 AI 调用审计落库。
7. 不实现用户额度和 token 计费。
8. 不实现生产级 API key 管理后台。
9. 不实现服务端加密存储或团队级密钥管理。
10. 不允许将真实 API key 写入代码或提交到 Git。
11. 不要求支持所有 provider。
12. 不要求实现复杂模型能力矩阵。
13. 不删除旧的 `apps/web/src/components/ai/ai-assistant-panel.tsx`，但本阶段不在文档页重新接入该右侧面板。

本阶段的重点是让编辑器内 AI 助手可以真实调用模型并验证效果。

## 4. 用户与使用场景

### 4.1 目标用户

本阶段面向开发者、课程学习者和后续维护者。

开发者需要通过本阶段明确：

1. `ai-workspace` 定义 AI 动作。
2. `editor` 负责把 AI 动作接入 BlockNote 编辑器交互。
3. `apps/web` 的 API route 负责真实模型调用。
4. API key 由用户在设置页「AI 配置」中录入，并只用于发起真实模型请求。

### 4.2 核心场景

#### 场景一：选区润色

1. 用户在文档中选中一段文字。
2. 格式化工具栏出现 AI 按钮。
3. 用户点击 AI 按钮。
4. 用户选择「润色」。
5. AI 返回润色后的内容。
6. 用户接受或撤销修改。

#### 场景二：slash menu 使用 AI 动作

1. 用户在编辑器中输入 `/`。
2. slash menu 展示 BlockNote 默认块和 AI 动作入口。
3. 用户选择「AI 写作」或具体 AI 动作。
4. AI 菜单打开。
5. 用户输入或选择 prompt。
6. AI 生成内容并写入文档。

#### 场景三：真实模型连通测试

1. 用户打开设置页的「AI 配置」。
2. 用户选择 provider，填写 model、baseUrl 和真实 API key。
3. 用户回到文档编辑页。
4. 用户使用 AI 动作。
5. 确认服务端 route 能基于设置页配置调用真实模型并流式返回结果。

## 5. 架构定位

### 5.1 包职责

推荐边界如下：

```text
packages/ai-workspace
  定义 AI 动作、prompt、上下文、请求协议和配置类型

packages/editor
  接入 @blocknote/xl-ai，渲染编辑器内 AI 菜单、slash menu、toolbar AI 按钮，并把 ai-workspace action 映射为编辑器 AI 入口

apps/web
  提供设置页 AI 配置 UI 和服务端 API route，使用用户配置调用真实模型

packages/rag
  后续提供真实知识检索结果，本阶段不接入
```

### 5.2 与 blocknote-demo 的关系

`blocknote-demo` 中的以下实现可以参考：

1. `BlockNoteAIMenu.tsx`。
2. `EditorAIMenuController.tsx`。
3. `BlockNoteSlashMenu.tsx`。
4. `EditorFormattingToolbar.tsx`。
5. `server/routes/chat.js`。
6. `server/utils/chatTransport.js`。

但不能直接整体照搬：

1. demo 使用 Vite，当前项目使用 Next.js App Router。
2. demo 使用独立 Express 服务，当前项目优先使用 `apps/web` API route。
3. demo 的 React、TypeScript、Vite 版本与当前项目不同。
4. demo 中的 prompt 和 AI action 需要替换为 `packages/ai-workspace` 的定义。
5. demo 的环境配置方式需要改为当前项目的设置页配置方式。

## 7. 产品需求

### 7.1 编辑器内 AI 菜单

`packages/editor` 需要提供编辑器内 AI 菜单。

能力要求：

1. 支持展示 `ai-workspace` 动作分类。
2. 支持展示具体 AI 动作。
3. 支持用户手动输入 prompt。
4. 支持显示 AI 生成状态。
5. 支持 AI 结果确认、撤销或重试。

验收标准：

1. 用户可以在编辑器内打开 AI 菜单。
2. AI 菜单能展示写作、翻译、文档、研究类动作。
3. 菜单动作来自 `packages/ai-workspace`，不在 `editor` 中重复硬编码业务 prompt。

### 7.2 选区 AI 按钮

选中文本后，格式化工具栏需要展示 AI 按钮。

能力要求：

1. 用户选中文本后可以点击 AI 按钮。
2. AI 菜单应定位到选区附近。
3. AI 动作默认使用当前选区作为上下文。
4. 对 `requiresSelection` 的 action，需要在无选区时禁用或不展示。

验收标准：

1. 选中文本后可以看到 AI 入口。
2. 点击 AI 入口后能打开 AI 菜单。
3. AI 请求包含当前选区内容。

### 7.3 Slash Menu AI 入口

slash menu 需要包含 AI 动作入口。

能力要求：

1. 输入 `/` 可以打开默认 slash menu。
2. slash menu 仍保留 BlockNote 默认块。
3. slash menu 额外展示 AI 动作入口。
4. 选择 AI 动作后打开 AI 菜单或直接触发对应 prompt。

验收标准：

1. 默认 BlockNote slash menu 功能不丢失。
2. AI 动作能从 slash menu 进入。
3. AI 动作入口来自 `packages/ai-workspace`。

### 7.4 AI 动作映射

`packages/editor` 需要将 `packages/ai-workspace` 的 action 映射成 BlockNote AI 可执行动作。

映射规则：

1. `action.title` 显示为菜单标题。
2. `action.description` 显示为菜单说明。
3. `action.prompt` 作为默认用户要求。
4. `requiresSelection` 用于控制选区场景。
5. `requiresRag` 本阶段可以展示为禁用或提示“后续接入知识库”。

验收标准：

1. `editor` 不重复维护第二份 AI action registry。
2. 修改 `ai-workspace` action 后，编辑器 AI 菜单可以同步变化。
3. `find-citations` 不应假装完成真实 RAG 检索。

### 7.5 DocumentEditor AI 扩展接口

`DocumentEditor` 需要暴露明确的 AI 扩展接口，避免 AI 配置、transport 路径和上下文构建逻辑散落在内部实现中。

推荐扩展 props：

```ts
export type DocumentEditorAIConfig = {
  enabled: boolean;
  api: "/api/ai/editor";
  settings: AIWorkspaceSettings;
  apiKey?: string;
};

export type DocumentEditorProps = {
  documentId: string;
  initialContent?: EditorDocumentContent;
  editable?: boolean;
  ai?: DocumentEditorAIConfig;
  buildAIContext?: () => AIWorkspaceDocumentContext;
  onAIError?: (error: Error) => void;
};
```

说明：

1. `ai.enabled` 控制是否启用编辑器内 AI。
2. `ai.api` 本阶段固定为 `/api/ai/editor`。
3. `ai.settings` 来自设置页当前 AI 配置。
4. `ai.apiKey` 来自设置页录入的 API key，本阶段用于课程演示测试。
5. `buildAIContext` 用于补充文档标题、选区、光标附近内容等上下文。
6. `onAIError` 用于页面统一展示或记录 AI 错误。

验收标准：

1. `DocumentEditor` 不在内部硬编码 provider、model、baseUrl 和 API key。
2. `DocumentEditor` 不在内部硬编码 API route 以外的业务配置。
3. `apps/web` 可以把设置页当前 AI 配置传给 `DocumentEditor`。
4. 关闭 `ai.enabled` 后，基础编辑器能力不受影响。

### 7.6 文档上下文 Adapter

`packages/editor` 需要提供 BlockNote 到 `AIWorkspaceDocumentContext` 的 adapter。

能力要求：

1. 从当前文档中提取可读文本。
2. 从当前选区中提取 `selectedText`。
3. 从光标附近内容中提取 `surroundingText`。
4. 保留 `documentId` 和文档标题。
5. 不在 adapter 中调用模型。
6. 不在 adapter 中调用 RAG 检索。

验收标准：

1. AI 请求前能构造 `AIWorkspaceDocumentContext`。
2. `ai-workspace` 不依赖 BlockNote 类型。
3. BlockNote 相关转换逻辑集中在 `packages/editor`。
4. action 定义和真实调用之间不缺少上下文装配层。

### 7.7 真实模型调用

本阶段需要接入真实模型调用，用于测试 AI 助手效果。

要求：

1. API key 由用户在设置页「AI 配置」中录入。
2. 前端只调用相对路径 API route，不直接调用模型供应商接口。
3. 服务端 route 使用 AI SDK 或 BlockNote 官方服务端 transport。
4. 服务端 route 返回 BlockNote AI 所需的流式响应。
5. 至少支持一个真实 provider。
6. API key 不得出现在 URL、源码、构建产物或服务端日志中。

推荐设置页字段：

```text
provider=deepseek
model=deepseek-chat
baseUrl=https://api.deepseek.com
apiKey=
temperature=0.7
maxTokens=2048
```

如果使用 OpenAI-compatible provider，可以继续使用：

```text
provider=custom
model=
baseUrl=
apiKey=
```

验收标准：

1. 设置页可以录入 provider、model、baseUrl 和 API key。
2. API key 以 password 输入框或等价方式展示，默认不可见。
3. 真实密钥不会被提交到 Git。
4. API key 不通过 URL query、日志或错误信息暴露。
5. AI 请求可以返回真实模型结果。

### 7.8 服务端 AI Route

`apps/web` 需要提供服务端 route 处理 BlockNote AI 请求。

能力要求：

1. 校验请求体中是否包含 messages。
2. 校验是否包含 BlockNote tool definitions。
3. 创建模型 provider。
4. 创建 BlockNote server transport。
5. 将 UI message stream 返回给前端。
6. 失败时返回明确错误信息。

验收标准：

1. 缺少 messages 时返回 400。
2. 缺少 tool definitions 时返回 400。
3. 模型调用失败时返回 502 或可诊断错误。
4. 成功时返回流式响应。

## 8. 安全与设置页配置要求

### 8.1 API Key 管理

本阶段需要真实 API key，配置入口是设置页「AI 配置」。

要求：

1. 设置页提供 API key 输入框。
2. API key 输入框默认使用 password 模式。
3. 本阶段固定使用浏览器 localStorage 保存设置页 AI 配置，包含 API key，用于课程演示和本地测试。
4. 服务端 route 只使用本次请求携带的 API key 调用模型，不将 API key 写入日志。
5. `packages/ai-workspace` 不包含真实 API key。
6. `packages/editor` 不包含真实 API key。
7. 前端 bundle 不包含任何预置真实 API key。
8. 生产级方案应在后续阶段替换为服务端加密存储或密钥管理服务。
9. 页面必须明确提示：localStorage 保存 API key 仅用于本地课程演示，不适合作为生产方案。

验收标准：

1. `rg "sk-"` 不应在源码中搜到真实密钥。
2. API key 不出现在 Git 追踪文件中。
3. API key 不出现在 URL、错误页面或服务端日志中。

### 8.2 设置页关系

当前设置页的 AI 配置需要承担真实模型测试配置入口。

本阶段要求：

1. 设置页可以展示并编辑 provider、model、baseUrl、temperature、maxTokens、enabledActions。
2. 设置页可以录入真实 API key。
3. 设置页需要提示 API key 仅用于本地测试或当前工作台配置。
4. 编辑器 AI 请求应使用设置页当前 AI 配置。
5. 设置页配置本阶段固定使用 localStorage 保存，后续再迁移到服务端加密存储。

## 9. 工程需求

### 9.1 依赖管理

推荐新增依赖：

```text
@blocknote/xl-ai
ai
@ai-sdk/openai-compatible
```

说明：

1. `@blocknote/xl-ai` 用于编辑器内 AI Extension 和菜单。
2. `ai` 用于前后端 AI stream 协议。
3. `@ai-sdk/openai-compatible` 用于 DeepSeek 或其他 OpenAI-compatible provider。
4. 依赖版本需要与当前 React 18、Next.js 14、TypeScript 5.4 兼容。
5. 不为接入 demo 而升级到 React 19 或 TypeScript 6。

验收标准：

1. `packages/editor` 管理 `@blocknote/xl-ai` 前端编辑器集成依赖。
2. `apps/web` 管理 `ai` 和 `@ai-sdk/openai-compatible` 等服务端 route 所需依赖。
3. `@blocknote/xl-ai/server` 只能在服务端 route 中通过 server-only 路径使用，不能进入客户端组件依赖链。
4. 不引入 Express。
5. 不引入独立 server 应用。

### 9.2 Next.js 客户端与服务端边界

要求：

1. BlockNote AI UI 组件必须是客户端组件。
2. 模型调用必须在服务端 route 中执行。
3. 前端只知道 API route 路径。
4. 前端不读取 `process.env.AI_API_KEY`，而是使用设置页中的用户配置调用同源 API route。
5. 前端 transport 固定调用 `/api/ai/editor`。

验收标准：

1. `packages/editor` 中的 AI UI 可以正常在文档页挂载。
2. `apps/web` route 可以读取请求中的 AI 配置，并只在服务端执行真实模型调用。
3. 构建时不出现浏览器 API 在服务端执行的问题。
4. 服务端 SDK 不进入前端 bundle。

### 9.3 与 ai-workspace 的关系

要求：

1. AI 菜单动作来自 `packages/ai-workspace`。
2. prompt 基础内容来自 `packages/ai-workspace`。
3. `editor` 可以在必要时补充选区和光标上下文。
4. `ai-workspace` 不直接依赖 BlockNote。

验收标准：

1. `packages/ai-workspace` 仍是纯 TypeScript 领域包。
2. `packages/editor` 负责 BlockNote 适配。
3. `apps/web` 负责服务端 route。

4. 选中文本后 AI 按钮出现。
5. AI 菜单可以打开。
6. 选择「润色」或「改写」能发起真实请求。
7. AI 结果能回写到文档。
8. 用户可以接受或撤销修改。

### 10.3 安全验证

需要验证：

1. 前端源码无真实 API key。
2. Git 追踪文件中无真实 API key。
3. API key 不出现在 URL query 中。
4. 服务端错误不会把完整密钥输出到页面或日志中。
5. 设置页 API key 默认以 password 模式展示。

## 11. 非功能需求

### 11.1 可维护性

AI 编辑器 UI、AI 业务定义和服务端模型调用必须分层清晰，避免后续难以替换 provider 或升级 BlockNote。

### 11.2 可扩展性

本阶段完成后，应能继续扩展：

1. RAG 检索结果注入 prompt。
2. citation 插入文档。
3. AI 操作审计。
4. token 额度和成本统计。
5. 多 provider 切换。
6. 基于设置页配置选择模型。

### 11.3 用户体验

1. AI 菜单不应遮挡主要编辑内容。
2. AI loading 状态要可见。
3. AI 错误状态要可诊断。
4. 失败时不能破坏原文档内容。
5. 取消或撤销需要符合 BlockNote 默认交互。
